# Agent 与工具调用入门

## 1. 先区分三个概念

大模型应用里经常把 Tool Calling、Workflow、Agent 混在一起，其实它们不是一回事。

| 类型 | 特点 | 适合场景 |
|---|---|---|
| Tool Calling | 模型根据需要调用一个明确工具 | 查订单、查天气、计算价格 |
| Workflow | 程序写死步骤，模型在步骤中完成语言任务 | 报告生成、合同审查、资料整理 |
| Agent | 模型自己判断下一步，并多轮调用工具 | 复杂检索、自动排查、开放式任务 |

初学者建议：

1. 先做普通 Prompt。
2. 再做 RAG。
3. 再做工具调用。
4. 最后做 Agent。

## 2. 工具调用解决什么问题

模型擅长理解语言，但不擅长直接操作外部系统。工具调用让模型能够：

- 查询数据库。
- 调用订单 API。
- 搜索网页或内部文档。
- 计算价格和日期。
- 生成文件。
- 发起工作流。

例子：

用户问：“我的订单什么时候到？”

模型不应该编答案，而应该调用：

```text
get_order_status(order_id)
```

拿到真实物流状态后再回复用户。

## 3. 工具设计原则

工具要小而清楚：

- 一个工具只做一件事。
- 参数要明确。
- 返回值要结构化。
- 失败要有错误码。
- 权限要在工具层控制。

坏工具：

```text
do_everything(input)
```

好工具：

```text
search_policy(query, department)
get_order_status(order_id)
calculate_refund(order_id, reason)
create_support_ticket(user_id, summary)
```

## 4. 工具调用 Prompt

```text
你是一个客服助手。

规则：
1. 如果用户询问订单、物流、退款，必须先调用对应工具。
2. 不要编造订单状态。
3. 工具返回失败时，向用户说明需要人工处理。
4. 回复要简洁。

可用工具：
- get_order_status(order_id)
- get_refund_policy(product_type)
- create_ticket(user_id, issue)
```

## 5. Workflow 比 Agent 更适合多数业务

很多业务并不需要 Agent 自主行动，只需要固定流程。

例子：行业报告生成 Workflow

1. 解析资料。
2. 提取要点。
3. 检索补充资料。
4. 生成大纲。
5. 分章节写作。
6. 做事实核查。
7. 输出报告。

这些步骤由程序控制更稳定，模型只负责每一步里的语言任务。

## 6. Agent 适合什么

适合：

- 任务步骤不固定。
- 需要边做边观察结果。
- 需要多次调用不同工具。
- 用户目标开放，比如“帮我分析这个系统为什么报错”。

不适合：

- 高风险交易。
- 权限边界不清楚。
- 简单问答。
- 只需要固定流程。

## 7. Agent 风险

- 乱调用工具。
- 花费过多 token。
- 循环卡住。
- 被提示词注入诱导。
- 操作越权。
- 中间步骤难以调试。

控制方法：

- 限制最大步骤数。
- 每个工具做权限校验。
- 高风险操作需要人工确认。
- 记录每一步计划摘要和工具结果。
- 给工具设置超时和重试。

## 8. 从 0 到 1 练习

练习 1：做一个计算器工具。

- 用户用自然语言输入问题。
- 模型判断是否需要计算。
- 调用 calculator。
- 返回计算结果。

练习 2：做一个订单查询助手。

- 用户输入订单号。
- 调用 mock_order_api。
- 按状态回复。

练习 3：做一个资料分析 Agent。

- 工具 1：搜索知识库。
- 工具 2：总结文档。
- 工具 3：生成报告大纲。
- 限制最多 5 步。

## 9. 工具调用验收清单

- [ ] 工具参数清楚。
- [ ] 工具返回 JSON。
- [ ] 错误码可读。
- [ ] 权限在工具层校验。
- [ ] 模型不能绕过工具编造结果。
- [ ] 高风险操作需要确认。
- [ ] 有日志记录工具调用链路。
