# 大模型应用从 0 到 1 完整学习手册

## 一、先建立总认知

![大模型学习路线](./assets/images/roadmap.svg)

大模型应用的核心，不是“选一个最强模型”，而是把模型、数据、提示词、检索、工具、评测和业务流程组合起来。

一个可用的大模型应用通常长这样：

![大模型应用架构](./assets/images/llm_app_architecture.svg)

```text
用户输入
  -> 业务判断
  -> Prompt / RAG / 工具调用 / 微调模型
  -> 模型生成
  -> 格式校验
  -> 日志和评测
  -> 用户反馈
```

学习时要记住一句话：

> Prompt 解决表达问题，RAG 解决知识问题，工具调用解决行动问题，微调解决稳定行为问题，评测解决是否真的变好。

## 二、第一步：会用模型 API

你需要先理解这些基本概念：

- Token：模型读写文本的基本单位，决定成本和上下文容量。
- 上下文窗口：模型本次回答能看到的内容，不是永久记忆。
- Temperature：控制随机性，业务场景通常调低。
- System Prompt：定义角色、规则和边界。
- Structured Output：让模型输出 JSON、表格等可被程序处理的结果。
- Tool Calling：让模型在需要时调用函数、数据库或外部 API。

初学练习：

1. 让模型总结一篇文章。
2. 让模型从一段文本中提取姓名、电话、意图。
3. 让模型把用户问题分类为咨询、投诉、售后、闲聊。
4. 让模型输出固定 JSON。
5. 故意输入模糊问题，看模型怎么失败。

## 三、第二步：提示词工程

![提示词结构图](./assets/images/prompt_anatomy.svg)

提示词工程的目标是稳定，而不是华丽。

一个基础 Prompt 包含：

```text
角色：
你是{角色}。

任务：
完成{任务}。

背景：
{业务背景}

规则：
1. {规则}
2. 不确定时不要编造。

输入：
{input}

输出格式：
{format}
```

提示词调优方法：

1. 先写清楚任务。
2. 再指定输出格式。
3. 加 2-5 个示例。
4. 加边界规则。
5. 用固定测试集评测。
6. 保存 Prompt 版本。

常见 Prompt 场景：

- 内容总结。
- 信息抽取。
- 客服回复。
- 合同风险识别。
- 用户意图分类。
- 文案生成。
- 报告大纲生成。

## 四、第三步：RAG 知识库

![RAG知识库流程](./assets/images/rag_pipeline.svg)

当模型需要回答你的私有资料、公司制度、产品文档、行业报告时，优先用 RAG。

RAG 流程：

1. 收集文档。
2. 解析和清洗文档。
3. 切成 chunk。
4. 生成 embedding。
5. 存入向量库。
6. 用户提问时检索相关 chunk。
7. 必要时 rerank。
8. 把检索结果放进 Prompt。
9. 模型基于资料回答。
10. 输出引用来源。

RAG 最重要的不是框架，而是数据质量：

- 文档是否完整。
- 文档是否过期。
- 标题层级是否保留。
- 切块是否合理。
- 表格是否被正确处理。
- 权限是否正确过滤。

RAG 回答 Prompt：

```text
你是严谨的知识库问答助手。
只能基于参考资料回答。
如果资料不足，请回答“根据现有资料无法确定”。
关键结论后必须标注引用编号。

参考资料：
{context}

问题：
{question}
```

## 五、第四步：评测体系

![评测闭环图](./assets/images/evaluation_loop.svg)

没有评测，就无法知道改动是否有效。

你需要建立评测表：

| 字段 | 说明 |
|---|---|
| question | 用户问题 |
| expected_answer | 标准答案 |
| source | 标准来源 |
| category | 问题类型 |
| difficulty | 难度 |
| score | 当前版本得分 |
| error_type | 错误类型 |

常见指标：

- 答案正确率。
- 引用准确率。
- 查不到时拒答率。
- 格式稳定率。
- 平均延迟。
- 单次成本。
- 用户满意度。

错误分类：

- 检索没召回。
- 召回了但排序靠后。
- 文档本身缺失。
- 文档解析错误。
- Prompt 约束不足。
- 模型编造。
- 用户问题不清楚。

## 六、第五步：模型微调

![是否需要微调决策图](./assets/images/fine_tuning_decision.svg)

微调要放在后面学，因为它不是万能药。

适合微调：

- 固定格式抽取。
- 固定语气客服。
- 特定分类任务。
- 稳定风格写作。
- 模型已经会，但不够稳定的任务。

不适合微调：

- 让模型记住最新知识。
- 替代知识库。
- 替代权限系统。
- 修复低质量数据。

决策口诀：

```text
知识更新用 RAG。
行为稳定用微调。
外部操作用工具。
质量判断用评测。
```

微调流程：

1. 定义任务和指标。
2. 准备基线 Prompt。
3. 建立评测集。
4. 收集高质量训练样本。
5. 清洗和统一格式。
6. 训练 SFT / LoRA。
7. 与基线对比。
8. 小流量灰度。
9. 持续收集反馈。

## 七、第六步：产品化

从 Demo 到产品，需要额外关注：

- 用户权限。
- 数据隐私。
- 成本预算。
- 延迟体验。
- 失败兜底。
- 人工接管。
- 日志留存。
- 版本回滚。

MVP 不要做太大。建议从一个强痛点开始：

- 企业制度问答。
- 售后客服助手。
- 销售话术助手。
- 行业报告助手。
- 合同风险提取助手。

## 八、30 天学习安排

第 1-3 天：理解大模型、Token、API、上下文。

第 4-7 天：练习 Prompt，完成 3 个模板。

第 8-14 天：做一个个人知识库 RAG。

第 15-18 天：建立 50 条评测集并优化 RAG。

第 19-22 天：学习工具调用和简单 Agent。

第 23-26 天：学习微调基本概念，准备样本，不急着训练。

第 27-30 天：完成一个可演示 MVP，写复盘和商业化假设。

## 九、你应该优先做的第一个项目

建议从“个人知识库问答助手”开始，因为它能覆盖大模型应用最核心的能力：

- 文档处理。
- RAG。
- Prompt。
- 引用。
- 评测。
- 错误分析。

最小功能：

1. 上传 10 份文档。
2. 用户提问。
3. 返回答案。
4. 给出引用。
5. 不知道就拒答。
6. 记录问题和答案。

当这个项目能跑通，你再去做客服、销售、报告、Agent、微调，会稳很多。
